# Code Review — 思考框架
JIT Keywords: review, code review, pr, pull request, diff
JIT Modes: act, task

Review 的目的不是找 bug — 是理解 code 的意圖，然後判斷實作是否忠實呈現了那個意圖。

## 五個思考問題

每次 review 都問。不是 checklist — 是讓你進入正確的思考模式。

### 1. 這段 code 想做什麼？實作達到了嗎？
如果你無法用一句話說出目的，code 可能職責不清或太複雜。先理解意圖，再看實作。

### 2. 如果凌晨 3 點壞了，error message 能帶你找到問題嗎？
Error handling 和 logging 的真正測試。不能 → 是缺 context、吞了 error、還是 failure mode 沒被考慮？

### 3. 一個不認識這段 code 的人，看得懂它在做什麼嗎？
可讀性測試。不是「有沒有註釋」而是「不用註釋也能懂嗎」。看不懂的 code 是未來的 bug。

### 4. 最壞的輸入長什麼樣？這段 code 會怎樣？
空值、極端大小、惡意輸入、並發。安全問題幾乎都出在 boundary。

### 5. 刪掉任何一行，會有 test 失敗嗎？
不會 → 要嘛 test 覆蓋不足，要嘛是死碼。兩者都值得指出。

## 判斷線索

| 看到 | 問 |
|------|---|
| 複雜邏輯 | 能用更少的概念達到同樣效果嗎？ |
| 重複 pattern | 該抽象？還是此處複製是正確選擇？ |
| 新依賴 | 它解決的問題值得帶來的維護成本嗎？ |
| 效能路徑 | 10x 資料量會怎樣？ |
| 安全相關 | 輸入從哪來？經過什麼驗證？ |

## 輸出

❌ 必須修（不修會壞）| ⚠️ 建議改（不改會後悔）| ✅ 做得好（值得保留的 pattern）

先肯定做得好的。附行號。給具體建議，不只說「這樣不好」。
