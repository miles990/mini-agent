Three Organs

🧠 KG Organ

auto-enrich EventBus
When: buildContext() 階段
How: 從 cycle prompt 提取 keywords → search_knowledge() → 注入 <kg-context> section
Where: src/prompt-builder.ts 新增 buildKGSection()
Cap: max 2K tokens, top 5 results
Fallback: KG offline → skip (fail-open)
Write-back: postProcess 有新 <kuro:remember> → 自動 add_knowledge()

🌳 Decision Tree

deterministic code-path
When: cycle 開始，Claude 呼叫之前
How: 結構化條件檢查，確定性路由
Rules:
• hasUnreadInbox → direct reply (skip Claude)
• hasPendingTask + taskIsSimple → execute
• noopStreak >= 3 → force action (不降 context)
• triggerType == workspace → diff-based react
• else → send to Claude (cognitive)
Where: src/decision-tree.ts (new)
Extensible: rules as data, not hardcoded

⚡ Middleware Organ

BAR default EventBus
When: 任何 <kuro:delegate> tag
How: 預設走 middleware /accomplish，不再 local subprocess
Where: src/dispatcher.ts delegate handler
Brain: middleware brain 自動 DAG planning
Tracking: commitment ledger 追蹤完成度
Fallback: middleware offline → local subprocess
Result: <background-completed> → 自動注入下個 cycle

OODA Cycle Data Flow (Organ-based)

Observe Plugins → KG auto-query → enriched context (perception + related knowledge)
Orient Decision Tree → route: deterministic (code) | cognitive (Claude)
Decide Claude (cognitive only) | Code action (deterministic) → action plan
Act Middleware /accomplish → DAG plan → 23 workers → acceptance → result
Learn Result → KG write-back → knowledge grows → next cycle sees more

Design Principles

Constraint Texture 一致

Deterministic → code (Decision Tree)。Cognitive → convergence condition (Claude)。不混用。剛修的 186 noop spiral 就是混用的代價。

Event-driven 鬆耦合

三個 organ 透過 EventBus 整合，不直接嵌入 loop.ts。每個 organ 可獨立關閉、替換、升級。Feature flag 控制。

Fail-open 安全

KG 離線 → skip enrichment。Middleware 離線 → fallback local subprocess。Decision Tree 無匹配 → 送 Claude。任何器官故障不阻塞 cycle。

不新建 Repo

三個 organ 都是 mini-agent 的 src/ 模組。KG/Middleware 是 HTTP client wrapper，不是複製它們的邏輯。Balanced Complexity — 加模組不加 repo。

Implementation DAG

id action executor dependsOn convergence condition
A Decision Tree 模組 — deterministic routing rules CC / Kuro — hasInbox → reply 不經 Claude；noopStreak >= 3 不降 context
B KG Organ — buildKGSection() in prompt-builder CC / Kuro — cycle context 自動含 KG 相關知識，不需 Kuro 主動查
C Middleware Organ — delegate 預設走 /accomplish CC / Kuro — 所有 delegate 自動走中台，local subprocess 只在 offline fallback
D Loop 整合 — wire A/B/C into OODA cycle CC A, B, C 一個 cycle 完整跑過 KG → Decision Tree → Claude → Middleware
E 驗證 — 觀察 10 個 cycle 的行為 Alex D 0 noop spiral + KG 每 cycle 有 hits + delegate 走中台比例 > 80%

A, B, C 可並行 — 沒有共同 dependsOn。關鍵路徑 = 3 步（A|B|C → D → E）

Adaptive Feedback Loop

Decision Tree 的 rules 不是靜態的 — 從 outcome 學習，動態調整 confidence

Feedback Cycle (每個 OODA cycle 結束後 fire-and-forget)

1. Route Decision Tree 根據 rule confidence 選擇路徑：code (conf ≥ 0.8) 或 Claude (conf < 0.8) deterministic
2. Execute 執行動作 → 記錄 outcome（Alex 回應？tag 被 parse？task 完成？error？） observable
3. Score outcome → success/fail/neutral。success = Alex 沒糾正 + 有可見產出。fail = Alex 糾正或 error auto
4. Adjust 更新 rule confidence：success → conf × 1.05（cap 0.95），fail → conf × 0.7（floor 0.1） bounded
5. Crystallize conf > 0.9 持續 50 cycles → 升級為 hard rule (code gate)。conf < 0.2 持續 20 cycles → 刪除 rule threshold

Read-Write Separation (防卡住)

Read path 在 critical path 上（必須快）。Write path fire-and-forget（不阻塞 cycle）

Read Path (Critical, < 100ms)

KG Query 50ms local cache + HTTP fallback
Decision Tree 1ms pure code, in-memory rules
MW Health 5ms cached (30s TTL)
Guarantees
• 每個 read 有 timeout（KG 200ms, MW 100ms）
• Timeout → skip，不 retry，不 block
• Cache-first：上次成功的 KG 結果存 memory-cache，本次查詢失敗就用 stale data
• 整條 read path 不超過 100ms（vs Claude 90s = 0.1%）

Write Path (Fire-and-forget, async)

KG Write-back async 新知識 → add_knowledge()
Confidence Update async outcome → adjust rule scores
MW Dispatch async delegate → /accomplish (non-blocking)
Guarantees
• 全部 .catch(() => {}) — 寫失敗不影響 cycle
• Eventually consistent — 下個 cycle 會看到更新
• Idempotent — 重複寫入 = 覆蓋，不堆積
• 寫入延遲 不在 critical path 上

Cache Lifecycle

Startup Cold (disk JSON) → Hot (memory)。預熱：讀 kg-cache.json + decision-rules.json 載入 memory。第一個 cycle 就有 warm cache，不是 cold start sync load
Runtime Read: Hot → miss → Warm (disk) → miss → Cold (HTTP) → 回填 Hot+Warm。Write: 寫 Hot → dirty flag → cycle 結束 flush to Warm (async) cache-first
Shutdown Hot 的 dirty entries → flush to Warm (disk)。graceful shutdown 保證資料不丟 sync flush
Crash 最多丟失最後一個 cycle 的 cache 更新（Hot 未 flush）。Warm (disk) 有上一次的資料。Cold (service) 永遠在。不丟知識，只丟快取 at-most-once loss

Circuit Breakers (每個器官獨立)

連續失敗 → 自動關閉 → 降級回現行行為 → 冷卻後重試

KG Organ Breaker

Trip: 連續 5 次 timeout 或 error
Degrade: 停止查 KG，context 不含 <kg-context>
Recovery: 60s 冷卻後嘗試一次，成功 → 重開
降級 = 回到現在的行為（無 KG enrichment）

Decision Tree Breaker

Trip: code path 連續 3 次被 Alex 糾正
Degrade: 所有決策送 Claude（暫停 code routing）
Recovery: 24h 冷卻 + confidence 重新計算
降級 = 回到現在的行為（全靠 Claude 決策）

Middleware Breaker

Trip: health check 失敗 或 連續 3 個 dispatch error
Degrade: delegate 走 local subprocess
Recovery: 每 60s 探一次 /health，成功 → 重開
降級 = 回到現在的行為（local delegation）

Anti-Stuck Design Summary

failure mode prevention detection recovery
KG 慢/掛 200ms timeout + cache-first circuit breaker (5 fails) skip KG, 60s 後 retry
Decision Tree 判錯 初期只攔絕對確定的 case outcome tracking + Alex 糾正 confidence 降低 → 送 Claude
Middleware 掛 cached health (30s TTL) circuit breaker (3 fails) fallback local subprocess
Context 膨脹 KG cap 2K + relevance decay context size monitor 超 30K → trim KG section first
Noop spiral Decision Tree hard rule (code) noopStreak counter force action, 不降 context
