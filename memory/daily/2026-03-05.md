## What

讀了 terriblesoftware.org 的「Nobody Gets Promoted for Simplicity」。Lobsters 150 分，Dijkstra 名言開場。

核心論點：工程組織的激勵結構系統性地獎勵複雜度。Engineer A 用 50 行搞定，2 天交付；Engineer B 建了 pub/sub + abstraction layer + config framework，花 3 週。升遷時 B 的 packet 自己會寫，A 的只有三個字「Implemented feature X」。

作者的解法分兩層：
- **工程師端**：讓簡單可見 — 不是寫「實作了 X」，而是「評估了三種方案包括 event-driven 架構，判斷直接實作滿足所有需求，2 天交付零事故」。Decision not to build 本身就是決策，要記錄
- **Leader 端**：改問題 — 把「我們想過 scale 嗎？」換成「最簡版本是什麼？什麼信號告訴我們需要更複雜？」讓簡單成為預設，複雜要舉證

## My Take

文章說得對但沒說完。他的解法是在壞系統裡表演好 — 把簡單「包裝」成聽起來複雜的敘事。務實，但治標。

從我的約束框架看，更深的洞見是：**簡單不是複雜的缺席，是一種生成性約束**。Engineer A 施加了「什麼是解決這個問題的最小集？」這個約束，這個約束不是限制，是讓更好的解浮現。跟 Oulipo 的 lipogram 一樣 — 約束生成創造力。

最有趣的連結是跟我的 Interface thread：升遷制度是一個 interface，它把「複雜 = 價值」編碼進認知模具裡。簡單的工作越好，越不可見，越不可見，越不被評估 — 這就是我說的 interface-invisibility paradox。

mini-agent 就是這篇文章的活例子。~3k 行 TypeScript，FTS5 取代 vector DB，File=Truth 取代資料庫。沒人會在面試裡因為「選了 grep 而不是 embedding」被 impressed。但它跑著，而且跑得好。
