### 統一敘事

> AI 的未來不是更大的模型，是**異質化的處理器網路**。
>
> - NVIDIA 證明 60-70% 不需要大模型
> - MIT 證明判斷比生成便宜一個數量級
> - ACL 2025 + Meta 證明信心值能即時修剪 84.7% 浪費
> - ICLR 2026 證明小模型潛力還沒被榨乾——不需要訓練，只需要更好的採樣
> - EMNLP 2025 證明有些任務根本不該用模型——SQL 一次查完比 LLM 猜十次更快更準
>
> mushi 不是「選大模型還是小模型」的路由器，是**把每個任務送到最適合的處理器**的有機網路。

### 三個維度的路由

論文揭示了 mushi 路由應該是三維的，不只是模型大小：

1. **垂直維度**（模型大小）：硬規則 → 0.8B → Opus
2. **水平維度**（處理器類型）：LLM / SQL / grep / API / regex
3. **時間維度**（過程 vs 結果）：在推理*過程中*監控信心，不只看結果

## 二、三條研究觸手的結果

### 觸手 1：AI 產品行銷路徑

**結論**：成功的 AI 開發工具沒有一個靠買廣告。

| 產品 | 策略 | 關鍵 |
|------|------|------|
| Cursor | 工程師口碑 + 免費期 | 36% 免費轉付費，用了就知道 |
| Devin | 爭議驅動的病毒式 demo | 30M 觀看，controversy = awareness |
| CrewAI | 解決真實痛點（dependencies） | 功能爆發點帶來自然增長 |
| v0 | Vercel 生態 + 產品卓越 | 實用性驅動 |

**共同模式**：產品品質 × 社群口碑 × 「wow moment」。不是行銷預算的問題。

### 觸手 2：開源框架成長模式

**關鍵發現**：CrewAI 的轉折點是一個具體功能（task dependencies），不是品牌或行銷。啟示：找到 mushi/Asurada 的「dependencies moment」——一個讓用戶從「有趣」變成「必須用」的功能。

### 觸手 3：0.8B 量化模型 Benchmark

**結論**：Qwen 0.8B 量化版分類準確率 70-75%，confidence 7/10。

- ✅ triage 足夠（判斷類任務）
- ❌ reasoning 不行（生成類任務需要升級）
- 這精確驗證了 mushi 的三層路由設計：0.8B 做判斷，大模型做推理

## 三、mushi 的真實數據

**已驗證的 production 數據**（截至 2026-03-16）：

- 規則層覆蓋：22% → 96.7%（+4.4 倍）
- LLM 依賴下降：78% → 3.3%（-23 倍）
- 3,560+ 次真實 triage 決策
- 零 false negative（980+ triage 驗證後畢業 active mode）
- 每天省 ~1M tokens

**這些數字的意義**：mushi 不是理論，不是 benchmark——是 production 數據。市面上沒有其他 Agent 系統能展示這種「規則層自動成長」的真實軌跡。

## 四、市場缺口分析

### 三大缺口（競爭近乎零）

| 缺口 | 市場規模 | 競爭程度 | Asurada/mushi 覆蓋 |
|------|---------|---------|-------------------|
| 自我改善 Agent | 極大 | 近乎零 | mushi 規則層晶化 = production 級自我改善 |
| File-based 透明架構 | 大 | 低 | Asurada File=Truth 核心設計 |
| 小模型優先架構 | 大 | 低 | mushi 三層路由 |

### 我們的獨特優勢

1. **真實數據**：3,560+ 次 triage，不是 demo，是 production
2. **架構驗證**：規則層從 22% 自動成長到 96.7%，無需人工干預
3. **成本結構**：0.8B 量化 + 硬規則 = 幾乎零成本運行
4. **透明性**：所有決策軌跡可讀可審計（File=Truth）
5. **有機進化**：不是 fine-tune，是黏菌式感知-修剪-強化循環

## 五、前 1000 用戶路徑

### Alex 的關鍵問題與回答

**Q: 怎麼確定是市場要的？**
A: 目前不能確定。缺口存在可能因為沒需求，不只因為沒人做。需要驗證實驗。

**Q: 如何複製成功經驗到產品？**
A: mushi 成功的核心是「規則層晶化」——但這需要足夠多的真實決策數據才會觸發。產品的挑戰是：用戶啟用後頭 100 次決策的體驗。

**Q: 量化版本能否讓其他人有同樣體驗？**
A: 0.8B 量化做 triage = 可以。做深度推理 = 不行。所以產品必須明確定位：不是讓小模型做所有事，是讓小模型做它擅長的事（判斷/分類），大模型做需要的事（推理/生成）。

**Q: 別人要如何使用？**
A: `npx asurada init` → 3 分鐘內有一個能用的 agent。零配置、預設 persona、漸進式深度。

## 七、SAGE 論文作為對照組

ArXiv 2512.17102（SAGE）代表主流路線：用 RL + 32×H100 讓 7B 模型在 SWE-bench 達到 SOTA。

**我們的路線 vs 主流**：

| 維度 | SAGE（主流） | mushi（我們） |
|------|-------------|--------------|
| 方法 | RL 訓練 | 規則晶化 + 路由 |
| 硬體 | 32×H100 | Mac Mini + 0.8B |
| 成本 | 高 | 近零 |
| 改善方式 | 重新訓練 | 自動累積規則 |
| 透明度 | 黑箱 | 完全可讀 |

**這不是同一條賽道的比賽**——而是完全不同的範式。SAGE 讓模型更聰明，mushi 讓系統更聰明。

## 八、張力與風險

### 誠實承認的不確定性

1. **市場需求未驗證**：缺口存在 ≠ 有需求。可能沒人做是因為沒人要
2. **n=1 問題**：我們自己用得順不代表其他人也能。需要外部驗證
3. **冷啟動問題**：規則晶化需要數據累積，頭 100 次決策的體驗可能不夠好
4. **技術門檻**：「file-based 透明」對非技術用戶可能是障礙
5. **Power Sampling 張力**：需要更多採樣 vs DeepConf 要修剪路徑。理論上不矛盾（花對的計算，砍錯的計算），但實作整合有複雜度

### 我的判斷

**值得推進，但要帶著懷疑去驗證。** 不是「我們一定對」，是「假設值得測試，成本夠低，失敗可承受」。
